# tests

## 功能概述

`tests/` 目录包含 `app-server` crate 的集成测试。所有测试被聚合到一个单一的测试二进制文件中（通过 `all.rs` 入口），以减少编译时间并共享测试基础设施。

测试通过启动真实的 app-server 进程（以 stdio 模式），通过 JSON-RPC 协议与之交互来验证端到端的功能正确性。测试基础设施提供了 mock 模型服务器、配置生成、认证模拟等丰富的辅助工具。

## 目录结构

```
tests/
├── all.rs                      # 测试二进制入口，聚合所有测试模块
├── common/                     # 公共测试基础设施（作为独立 crate）
└── suite/                      # 测试用例集合
```

## 核心接口/API

### all.rs - 测试入口

- 仅包含 `mod suite;` 声明，将所有测试模块聚合到单一二进制中
- 这种模式避免了 Rust 集成测试为每个文件生成独立二进制的默认行为，显著减少编译和链接时间

### 测试执行模式

集成测试的典型流程：
1. 创建临时目录作为 `CODEX_HOME`
2. 写入测试配置文件（`config.toml`）
3. 可选：启动 mock 模型服务器
4. 启动 `McpProcess`（app-server 进程）
5. 发送 `initialize` 请求
6. 执行测试场景（发送请求/通知，验证响应/通知）
7. 关闭进程
